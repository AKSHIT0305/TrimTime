# TrimTime API — Database Layer

## Stack
- **Database**: PostgreSQL 15+
- **ORM**: Prisma 5 (with generated TypeScript client)
- **Language**: TypeScript / Node.js 22

---

## Running Migrations

### Development (recommended)

Creates the migration file, runs it, and regenerates the Prisma client:

```bash
npm run db:migrate:dev
# Prisma will prompt: "Enter a name for the migration" → type: init
```

### First time with an existing migration file

```bash
# Apply all pending migrations without prompting (used in CI/CD)
npm run db:migrate:deploy
```

### Check migration status

```bash
npm run db:status
# Shows which migrations have been applied
```

### Reset the database (dev only — destroys all data)

```bash
npm run db:migrate:reset
# Drops DB, recreates it, runs all migrations, then runs seed
```

## Generating the Prisma Client

After any schema change, regenerate the TypeScript client:

```bash
npm run db:generate
# Outputs to: src/generated/prisma/
```

> The client is output to `src/generated/prisma/` rather than `node_modules/` so it's
> version-controlled and visible in IDE intellisense without requiring a full install.

## Schema Overview

```
users
  └── salons (owner_id → users.id)
        ├── salon_photos  (salon_id → salons.id)
        ├── opening_hours (salon_id → salons.id)
        ├── services      (salon_id → salons.id)
        │     └── barber_services ←─┐
        ├── barbers ───────────────→┘ (many-to-many via barber_services)
        │     └── bookings (barber_id → barbers.id)
        └── bookings (salon_id → salons.id)
              └── queue_entries (booking_id → bookings.id, 1-to-1)

subscription_plans
  └── salons (plan_id → subscription_plans.id)

otp_codes (standalone — indexed by phone + expires_at)
```

### Key Design Decisions

| Decision | Reason |
|----------|--------|
| `slug` on salons | Clean public URLs: `/s/raj-barbers`. Unique, URL-safe. |
| Denormalized `customerName`, `serviceName`, `price` on bookings | Preserves historical data even if the user/service is deleted or prices change later. |
| `DayOfWeek` enum (not INT) | Self-documenting, prevents magic numbers in queries. |
| `UNIQUE(salonId, position)` on queue | Enforces at DB level that no two people share the same queue slot. |
| `is_open` + `open_time/close_time` per day | Supports partial-day closures and day-off toggles independently. |
| `ON DELETE CASCADE` for salon sub-resources | Deleting a salon cleans up photos, services, barbers, bookings atomically. |
| `ON DELETE SET NULL` for user references | Bookings and barbers survive if the user account is deleted (GDPR-friendly). |
| `updated_at` trigger in SQL | Guarantees auto-update even for raw SQL mutations that bypass Prisma. |

## Common Prisma Queries (Reference)

```typescript
import { prisma } from './src/lib/prisma';

// Get salon with all services + barbers
const salon = await prisma.salon.findUnique({
  where: { slug: 'raj-barbers' },
  include: {
    services: { where: { isActive: true }, orderBy: { sortOrder: 'asc' } },
    barbers:  { where: { isActive: true }, include: { barberServices: true } },
    openingHours: { orderBy: { dayOfWeek: 'asc' } },
  },
});

// Today's bookings for a salon
const today = new Date();
today.setHours(0, 0, 0, 0);
const tomorrow = new Date(today);
tomorrow.setDate(tomorrow.getDate() + 1);

const bookings = await prisma.booking.findMany({
  where: {
    salonId: salon.id,
    appointmentAt: { gte: today, lt: tomorrow },
    status: { not: 'cancelled' },
  },
  include: { barber: true, service: true },
  orderBy: { appointmentAt: 'asc' },
});

// Current queue for a salon (ordered by position)
const queue = await prisma.queueEntry.findMany({
  where:   { salonId: salon.id, servedAt: null },
  orderBy: { position: 'asc' },
  include: {
    booking: {
      include: { barber: { select: { id: true, name: true, photoUrl: true } } },
    },
  },
});

// Add to queue (inside a transaction to prevent position conflicts)
const entry = await prisma.$transaction(async (tx) => {
  const maxPosition = await tx.queueEntry.aggregate({
    where: { salonId: salon.id, servedAt: null },
    _max: { position: true },
  });
  const nextPosition = (maxPosition._max.position ?? 0) + 1;

  return tx.queueEntry.create({
    data: {
      salonId: salon.id,
      bookingId: booking.id,
      position: nextPosition,
      estimatedWait: nextPosition * 20, // placeholder; recalculated by service
    },
  });
});
```

## Production Deployment Checklist

- [ ] Set `DATABASE_URL` as an environment variable (never in code)
- [ ] Run `npm run db:migrate:deploy` (not `dev`) — no interactive prompts
- [ ] Do **not** run seed in production (or use a separate `seed:prod` with minimal data)
- [ ] Enable PostgreSQL SSL: append `?sslmode=require` to `DATABASE_URL`
- [ ] Set up a read replica and point `REPLICA_DATABASE_URL` for analytics queries
- [ ] Schedule a job to run `DELETE FROM otp_codes WHERE expires_at < NOW()` daily
- [ ] Back up the `migrations/` folder — it's the audit trail for your schema history
